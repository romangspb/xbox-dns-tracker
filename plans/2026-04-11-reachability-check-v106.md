## Спецификация (WHAT & WHY)

### Проблема
Пользователь (в РФ) не знает заранее, какой из 47 собранных DNS/xsts-методов реально сработает в его сети. Серверная проверка даёт статус, но она запускается из GitHub Actions (не из РФ) — её результаты не отражают реальный ISP пользователя. Сейчас пользователю приходится вручную вбивать DNS в Xbox, ждать подключения, смотреть есть ли ошибка, и так по кругу.

### Цель
Одна кнопка **«Проверить мой доступ к Xbox»** на сайте. За 2-5 секунд браузер пользователя проверяет доступность всех target IP (куда DNS перенаправляют xsts-запросы) из его реальной сети и показывает:
- Агрегат сверху: «Из 45 путей тебе доступны 14»
- Метки на каждой карточке (зелёная галочка / красный крестик / серое «не проверено»)
- Автофильтр «показывать только доступные»

### Ключевой инсайт
Серверный `checker.py` для рабочих DNS уже сохраняет `resolved_ip` в `data.json` — конкретный proxy IP, в который DNS разрешает `xsts.auth.xboxlive.com`. Если этот proxy IP недоступен из сети пользователя → DNS у него в Xbox не сработает. Если доступен → **вероятно** сработает (браузер не может проверить UDP/53, которые использует Xbox, но если HTTPS к target IP проходит, то и DNS должен пройти).

### Honest messaging
Браузер проверяет TCP/HTTPS reachability, а не реальный DNS-резолв на Xbox-консоли. Поэтому:
- ❌ «недоступен» = **точно** не сработает на Xbox (надёжный негативный сигнал)
- ✅ «доступен» = **вероятно** сработает (нужно попробовать на консоли)
- ❓ «не проверено» = данных не хватило (IPv6 или ошибка функции)

Эти формулировки дословно появятся в UI и меню помощи — не врём пользователю.

### Границы (что НЕ делаем)
- Не переписываем `checker.py` — остаётся как есть, кормит `data.json` ежедневным cron
- Не меняем формат `data.json` — обогащаем данные только на стороне браузера
- Не делаем полноценную проверку IPv6 DNS — остаётся «ipv6_unchecked» из v1.0.5
- Не добавляем авторизацию на `/api/resolve` — доверяем 5 пользователям + кэш защищает от спама
- Не трогаем Telegram-бот / Cloud Gaming / прочий бэклог

### Архитектура (Путь А — всё на Vercel)

```
xbox-dns-tracker.vercel.app
├── / (статика из site/)
│   ├── index.html
│   ├── app.js
│   ├── style.css
│   └── data.json  ← обновляется GitHub Actions ежедневно
└── /api/resolve?dns=X.X.X.X
    └── Node.js Serverless Function
        ├── dns.Resolver.setServers([dns])
        ├── .resolve4('xsts.auth.xboxlive.com')
        └── возвращает { resolved_ip, cached }
```

**Портабельность (на случай блокировки Vercel в РФ):**
- `api/resolve.js` — стандартная `(req, res) => {...}` подпись, работает в Vercel, Express, Fastify
- Никаких Vercel-специфичных фич (Edge Config, KV, Middleware)
- Кэш — in-memory (Map), допускаем частичные промахи при cold start
- Секретов нет, переносить нечего
- GitHub Actions workflow для GitHub Pages **не удаляем** — откат на `git reset --hard v1.0.5-stable` восстанавливает старый деплой
- Сценарий миграции на VPS расписан в `plans/migration-from-vercel.md` (создадим в Фазе 5)

### Acceptance criteria
1. Кнопка «Проверить мой доступ к Xbox» видна на главной сверху на iPhone Safari
2. При первом клике показывается модалка с предупреждением про VPN
3. После подтверждения запускается проверка, показывается progress
4. За ≤10 секунд завершается проверка всех 45 методов (47 минус 2 IPv6)
5. Показывается агрегат «Из N путей тебе доступны X»
6. На карточках появляются метки: зелёная галочка / красный крестик / серое «?»
7. Работает фильтр «только доступные» (новая кнопка в `.status-filters`)
8. Результаты кэшируются в localStorage на 1 час — повторный клик не дёргает функцию, если кэш свежий
9. Кнопка «Обновить сейчас» рядом с результатом — принудительно игнорирует кэш
10. Меню помощи `?` содержит раздел про эту функцию с honest-messaging
11. Если `/api/resolve` не отвечает — сайт не ломается, показываются результаты только для cached DNS + баннер «не все пути проверены»
12. `node --check` проходит на `api/resolve.js` и `site/app.js`
13. Сайт открывается по `https://xbox-dns-tracker.vercel.app/`

## Challenge Log

### Вопрос 1: Решает ли это проблему?
Да. Проблема: пользователь не знает какой DNS сработает в его сети. Решение: автоматическая проверка за одну кнопку. Честная дисклеймер «доступен ≠ гарантированно работает» добавлена в UI — не вводим в заблуждение.

### Вопрос 2: Самое эффективное решение?
Путь А (всё на Vercel) выбран после рассмотрения:
- **Путь Б (отдельно фронт и API)** — надёжнее при блокировке, но сложнее (два деплоя, CORS). Отклонён т.к. портабельный код + git как backup снижают риск.
- **Путь В (полный переезд чекера на Vercel)** — слишком большая переделка, риск сломать работающий `checker.py`. В бэклог.
- **Browser-based DoH** — большинство bypass DNS не имеют DoH endpoint. Невозможно.

Выбранное решение минимально инвазивное: `checker.py` не трогаем, данные не меняем, добавляем один endpoint + UI.

### Вопрос 3: Code for code's sake?
- ❌ Не переписываем `checker.py` — работает, трогать незачем
- ❌ Не меняем формат `data.json` — совместимость с v1.0.5
- ❌ Не добавляем авторизацию — 5 доверенных юзеров + кэш защитят лимиты
- ❌ Не делаем полноценный IPv6 — отдельная задача в бэклоге
- ❌ Не делаем Telegram-бот уведомления — v1.1

## Think ahead — возможные проблемы и их решения

| Проблема | Решение |
|---|---|
| Vercel блокируют в РФ | `*.vercel.app` сейчас работает; портабельный код + план миграции на VPS (Фаза 6) |
| Юзер спамит кнопку | localStorage кэш 1 час + in-memory кэш функции 24 часа = не дёргаем бэкенд |
| Cold start функции | Первый вызов медленнее, но это один раз в день на юзера, приемлемо |
| VPN пользователя исказит результат | Модалка-предупреждение перед первой проверкой + напоминание в help |
| `no-cors` fetch к HTTP (не HTTPS) | Пропускаем HTTP — опасно из HTTPS-контекста. Target IPs у нас все HTTPS-capable |
| Browser блокирует fetch к private IP | Наши target IPs все публичные, private фильтруются на этапе парсинга (v1.0.2) |
| GitHub Actions работают в main, мы в feature-ветке | data.json для feature берём из main (Vercel автоматом выкатит новый preview при пуше main) |
| Сразу после мёрджа в main GitHub Pages тоже обновится и сломается | GitHub Pages будет получать тот же `site/` — он не зависит от `/api`, поэтому кнопка проверки там просто не найдёт backend и graceful degrade'нет к «не все пути проверены». Приемлемо |
| Юзер нажал «обновить» во время проверки | AbortController отменит текущий запрос, запустит новый |
| Вышел из вкладки во время проверки | Browser не гарантирует завершение fetch в background — допускаем, результат не сохранится, юзер кликнет ещё раз |
