### 2. Самое эффективное решение?

**Альтернатива 1:** Единый regex-парсер для всех источников.
- Плюс: меньше кода.
- Минус: источники слишком разные (HTML vs markdown vs structured). Regex для HTML — хрупко.
- **Отклонено.**

**Альтернатива 2:** Scrapy framework.
- Плюс: встроенные retry, pipeline, middleware.
- Минус: overkill для 5 запросов. Тяжёлая зависимость, кривая обучения.
- **Отклонено.**

**Альтернатива 3 (выбрана):** Модульные парсеры (функция на источник) + общий pipeline.
- Плюс: просто, читаемо, легко добавлять/убирать источники.
- Минус: нет встроенных retry (но httpx умеет таймауты, retry в 5 строк).
- **Лучший баланс простоты и надёжности для нашего масштаба.**

### 3. Нет ли "кода ради кода"?

- Без абстрактных классов/интерфейсов — простые функции.
- Без ORM/моделей — dict → JSON.
- Без конфигурационных файлов — константы в коде (5 пользователей, не enterprise).
- Без CLI-аргументов (argparse) — один режим запуска.
