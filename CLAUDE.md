# Xbox DNS Tracker — навигация по проекту

Этот документ — карта проекта. Читай его в начале каждой сессии, чтобы понимать, где что лежит и как с этим работать.

## Что это за проект

**Xbox DNS Tracker** — автоматический трекер DNS и способов обхода региональных ограничений Xbox (ошибка 0x80a40401) для 5 пользователей.
Парсит источники (Telegram-каналы, GitHub, сайты), показывает актуальные способы на мобильной веб-странице с классификацией по сложности и персональным трекингом статусов.

Подробности — в `.business/INDEX.md`.

## Ключевые принципы проекта

1. **Простота** — минимум действий для пользователя, всё в пару кликов с iPhone
2. **Автоматизация** — парсинг и обновление без ручного труда
3. **Актуальность** — никаких мёртвых DNS, входной фильтр по давности (2 мес.), неактивные помечаются, при повторном появлении — флаг "перепроверить"

## Стек (после ресерча)

- **Парсер**: Python 3.12+ (httpx, BeautifulSoup4, dnspython)
- **Данные**: JSON-файлы в репозитории (data.json, statuses.json) — SQLite/Supabase overkill для 5 пользователей
- **Фронтенд**: статичный HTML/CSS/JS (без фреймворков)
- **Хостинг**: GitHub Pages + GitHub Actions
- **Будущее (v1.1)**: aiogram (Telegram-бот)

## Версии продукта

- **v0.1** — Парсер + нормализованные JSON-данные (дедупликация, классификация, фильтр по давности)
- **v0.2** — Автопроверка DNS (резолв xsts.auth.xboxlive.com — нестандартный IP = bypass работает)
- **v1.0** — Веб-страничка + GitHub Actions + персональный трекинг
- **v1.1** — Telegram-бот (уведомления + кнопка "Обновить сейчас")

## Как работать с проектом

- Перед любой задачей — прочитай этот файл, затем нужные документы из `.business/` и `plans/`.
- Если меняется бизнес-логика — обнови `.business/INDEX.md` и нужные файлы.
- Если меняется технический план — обнови соответствующий план в `plans/`.
- Если появляется новая важная папка или документ — обнови этот `CLAUDE.md`.

## ВАЖНО: план для каждой новой функции

Любая функция, которую мы создаём в любом чате, всегда оформляется планом в папке `plans/`.

Правила:

1. Один план = одна функция. Если план уже есть — работаем с ним.
2. Имя файла: `YYYY-MM-DD-название-функции.md`.
3. План делится на фазы. У каждой фазы статус `[ ]` или `[x]`.
4. Каждая фаза содержит ДВА обязательных блока: «Что изменится» (было → стало на языке пользователя, для приёмки) и «Итог реализации» (технический лог: файлы, решения, проблемы — заполняется при закрытии фазы). Подробности — в `plans/README.md`.
5. В конце плана — итоговый блок: реализован целиком или нет, что осталось.
6. Любой агент обязан актуализировать план после каждой сессии — отмечать фазы, заполнять итоги.

## ВАЖНО: завершение каждого чата

В конце каждой сессии записывай рефлексию в файл `.business/история/YYYY-MM-DD-краткое-название.md` (создавай папку при необходимости).

Формат:

1. Какая задача была поставлена.
2. Как я её решал.
3. Решил ли — да / нет / частично.
4. Эффективно ли решение, что можно было лучше.
5. Как было и как стало.
